# Jellify Desktop — Accessibility & Internationalization Research

Scope: `macos/` SwiftUI app, macOS 14+. Audit of current code (`PlayerBar.swift`,
`Sidebar.swift`, `LibraryView.swift`, `SearchView.swift`, `AlbumDetailView.swift`,
`LoginView.swift`, `MainShell.swift`, `TrackRow.swift`, `Artwork.swift`,
`Theme.swift`) shows: no accessibility labels/hints/values, raw `Text` / `Button`
with icon-only labels (SF Symbol names like `"shuffle"`, `"backward.fill"`), a
home-rolled `GeometryReader`+`Capsule` progress bar with no control semantics,
all strings hardcoded in English, no `.xcstrings` catalog, a custom Figtree font
used without `scaledFont(for:)`, hardcoded numeric sizes, brand colors fixed per
token with no high-contrast variant, and no `Localizable.xcstrings` resource at
all. This document enumerates the issues to close that gap.

References consulted: Apple's Accessibility Programming Guide for macOS, Human
Interface Guidelines > Accessibility, WWDC 2023 "Catch accessibility issues
early" (session 10132), WWDC 2023 "Discover String Catalogs" (session 10155),
WWDC 2022 "Writing for interfaces", Apple Music VoiceOver behavior, community
reports on AppleVis.com regarding Music.app and Spotify's VO rotor gaps,
W3C WCAG 2.2 AA contrast thresholds.

### Issue 1: Establish accessibility baseline audit with Xcode Accessibility Inspector
**Labels:** `area:macos`, `area:a11y`, `kind:polish`, `priority:p0`
**Effort:** S

Before adding fixes, produce a reproducible audit report. Open Xcode > Open
Developer Tool > Accessibility Inspector. Point it at a running debug build,
run the Audit panel against each top-level screen (Login, Library, Search,
Album Detail, empty-state Home, player bar with track playing, player bar

**Acceptance criteria**

- `macos/docs/a11y/README.md` documents the audit command, what issues are

- A CI job (or `Scripts/a11y-audit.sh`) can launch the app in a known state

**Manual test**: With Accessibility Inspector targeting the running app, the
Audit panel surfaces the current known issues (no-label buttons on
`PlayerBar`'s `iconBtn("shuffle")`, `iconBtn("repeat")`, the logout button in

### Issue 2: Give every icon-only Button a VoiceOver label
**Labels:** `area:macos`, `area:a11y`, `kind:feat`, `priority:p0`
**Effort:** M

Every `Button` with an SF Symbol child renders as `"shuffle"` or `"pause.fill"`
to VoiceOver — the raw symbol name, which is unacceptable for VO, Voice
Control ("Tap play.fill" doesn't work), and Switch Control. Files to fix:

- `PlayerBar.swift` — shuffle, previous, play/pause, next, repeat, like,
  queue, volume, AirPlay, the `rightControls` block.
- `Sidebar.swift` — Home, Library, Search, Favorites, Albums, Artists, logout.
- `TrackRow.swift` — row context-menu trigger, like toggle.
- `LibraryView.swift` — the view-mode toggle (grid/list).
- `SearchView.swift` — clear-search button.
- `AlbumDetailView.swift` — Play All, Shuffle, Save/like.
- `Artwork.swift` — if it's interactive (it currently isn't; mark
  `.accessibilityHidden(true)` when purely decorative).

Pattern:

```swift
Button(action: model.togglePlayPause) {
    Image(systemName: isPlaying ? "pause.fill" : "play.fill")
        .accessibilityHidden(true) // icon is decorative, button owns the label
}
.accessibilityLabel(isPlaying ? "Pause" : "Play")
.accessibilityHint(isPlaying ? "Pause the current track" : "Resume playback")
```

For state-dependent buttons (shuffle on/off, repeat off/all/one) set
`.accessibilityValue(_:)` so VO announces the *current state*, not just the
action. For `repeat` that's `"Off"`, `"Repeat All"`, `"Repeat One"`.

**Acceptance criteria**
- Enable VoiceOver (`Cmd+F5`). Tab through every interactive element on
  every screen. Every stop reads a human-readable label, never a symbol
  name. No element reads "button" with no label.
- Toggle shuffle → VO announces "Shuffle, on" then "Shuffle, off".
- Voice Control ("Show names") overlays labels like "Play" and "Next",
  never "play.fill".

### Issue 3: Make the progress bar an accessible Slider with scrub + value
**Labels:** `area:macos`, `area:a11y`, `kind:feat`, `priority:p0`
**Effort:** M

`PlayerBar.swift` lines 79–88 draw progress as a `GeometryReader` + two
`Capsule`s. It's unreachable from the keyboard, invisible to VoiceOver, and
has no scrubbing affordance. Replace with a real `Slider`:

```swift
Slider(
    value: Binding(
        get: { model.status.positionSeconds },
        set: { model.seek(toSeconds: $0) }
    ),
    in: 0...max(1, model.status.durationSeconds),
    step: 1
) {
    Text("Playback position")
}
.accessibilityValue(formatTime(model.status.positionSeconds) +
                    " of " +
                    formatTime(model.status.durationSeconds))
.accessibilityLabel("Playback position")
.accessibilityAdjustableAction { direction in
    switch direction {
    case .increment: model.seek(toSeconds: model.status.positionSeconds + 5)
    case .decrement: model.seek(toSeconds: model.status.positionSeconds - 5)
    @unknown default: break
    }
}
```

Apply the same pattern to the volume slider in `rightControls`.

**Acceptance criteria**
- Tab focuses the progress bar; left/right arrow scrubs by ±5s.
- VoiceOver announces "Playback position, 1 minute 24 seconds of 3
  minutes 42 seconds, adjustable".
- VO rotor "Adjust value" increments/decrements.
- Visual: Slider can be styled to match the current capsule look via
  `.sliderStyle(...)` or a custom `SliderStyle` — the fix must not regress
  the dark-theme aesthetic.

### Issue 4: Combine TrackRow sub-labels into a single VO announcement
**Labels:** `area:macos`, `area:a11y`, `kind:polish`, `priority:p1`
**Effort:** S

`TrackRow.swift` renders index, title, artist, and duration as separate
`Text`s. VO reads them as four stops per row, making navigation miserable on
a 200-track album. Wrap the row and combine:

```swift
HStack { … }
    .accessibilityElement(children: .combine)
    .accessibilityLabel(
        "\(track.indexNumber ?? 0). \(track.name), " +
        "\(track.artistName), " +
        "\(durationSpoken(track.durationSeconds))"
    )
    .accessibilityHint("Double tap to play")
    .accessibilityAddTraits(track.isPlaying ? [.isSelected, .playsSound] : [])
```

Use `DateComponentsFormatter` with `.spellOut` for the duration — "three
minutes forty-two seconds" — instead of `3:42`, which VO would read as
"three colon forty-two".

**Acceptance criteria**
- One VO stop per track row, announcing title, artist, duration.
- The currently-playing row has the `.isSelected` trait and reads "selected".
- Performance: combining labels is cheap; verify a 500-row album scrolls
  smoothly.

### Issue 5: Logical tab order + Shift-Tab reverse traversal
**Labels:** `area:macos`, `area:a11y`, `kind:feat`, `priority:p0`
**Effort:** M

On macOS, `Tab` moves between focusable controls. In SwiftUI, only some
views are focusable by default — `Button` yes, custom gesture-driven views
no. Audit and correct order: sidebar nav → primary content search/filter →
list of rows → player bar (play/pause first, then transport, then volume).
Use `.focusSection()` to group, `.focusable()` to opt-in custom views
(progress bar once it's a `Slider` this is free), and `.focusEffectDisabled()`
only where we deliberately swap to a custom focus ring.

System Settings > Keyboard > "Keyboard navigation" must be enabled for Tab
to reach non-text controls on macOS — document this in
`macos/docs/a11y/keyboard.md` for testers.

**Acceptance criteria**
- With keyboard navigation on, Tab cycles: Sidebar Home → Library → Search
  → Favorites → Albums → Artists → Logout → Content area first control
  → track rows → Player transport → progress → volume → (loop).
- Shift-Tab reverses the order.
- Every focused element shows a visible focus ring (default blue, or a
  themed pink — see Issue 6).

### Issue 6: Visible, themed focus ring
**Labels:** `area:macos`, `area:a11y`, `kind:polish`, `priority:p1`
**Effort:** S

The default macOS focus ring is system blue, which clashes with the deep
purple theme. Either accept the system ring (fine, Apple prefers this), or
define a custom ring via `.focusEffect(.automatic)` with a rectangle overlay
driven by `@FocusState`. Whatever we pick must:

1. Be at least 2 px thick, offset 2 px from the control.
2. Maintain ≥ 3:1 contrast against `Theme.bgAlt` (#140B30) — the existing
   `Theme.accentHot` (#FF066F) passes at ≈ 7.8:1.
3. Never be removed wholesale with `.focusEffectDisabled()` on a focusable
   control.

**Acceptance criteria**
- All focusable controls show a visible focus ring when focused via Tab.
- Ring color validated with Colour Contrast Analyser against the surface
  behind it.

### Issue 7: `@FocusState` for search autofocus and modal focus
**Labels:** `area:macos`, `area:a11y`, `kind:feat`, `priority:p1`
**Effort:** S

`SearchView.swift` currently uses a `TextField` with no explicit focus.
When a user hits ⌘F (to be added in Issue 25), the search field should
autofocus. Use:

```swift
@FocusState private var searchFieldFocused: Bool

TextField("Search tracks, albums, artists", text: $query)
    .focused($searchFieldFocused)
    .onAppear { searchFieldFocused = true }
    .onExitCommand { searchFieldFocused = false }
```

Also apply to `LoginView.swift` — autofocus the server URL field on first
load, advance to username on ⏎, then password, then submit.

**Acceptance criteria**
- Launch the app; LoginView has the server-URL field focused with a
  visible cursor.
- From the Library, ⌘F focuses the search field in SearchView.
- Esc clears focus.

### Issue 8: Dynamic Type — Figtree + scaledFont
**Labels:** `area:macos`, `area:a11y`, `kind:feat`, `priority:p0`
**Effort:** M

`Theme.font(_:weight:italic:)` returns a fixed-size `Font.custom` which
does *not* scale with the user's Text Size preference (System Settings >
Accessibility > Display > Text Size, or the Larger Text slider). Replace
with `.custom(name, size: baseSize, relativeTo: .body)` — that's the scaled
form introduced in iOS 14 / macOS 11.

```swift
static func font(_ size: CGFloat,
                 weight: Font.Weight = .regular,
                 relativeTo textStyle: Font.TextStyle = .body) -> Font {
    Font.custom(fontName(for: weight), size: size, relativeTo: textStyle)
        .weight(weight)
}
```

Then audit every call site and assign an appropriate text style:
- Track title / album title → `.headline`
- Artist / subtitle → `.subheadline`
- Durations, counts, footer meta → `.caption`
- Sidebar "DESKTOP" kicker → `.caption2`

**Acceptance criteria**
- System Settings > Accessibility > Display > Text Size set to +3:
  every label grows visibly; no clipping, no fixed-height containers
  truncate content (audit `PlayerBar`'s `height: 78` — may need to become
  `minHeight` or a wrapped layout).
- Same test at +5 (extreme): app stays usable, text wraps or truncates with
  an ellipsis, sidebar reflows.

### Issue 9: Dynamic Type safe layouts — reflow Player Bar and Sidebar

### Issue 10: Respect Reduce Motion for ambient animations
**Labels:** `area:macos`, `area:a11y`, `kind:feat`, `priority:p0`
**Effort:** S

Any current or planned animation — the ambient purple wash on LoginView,
a spinning vinyl on AlbumDetailView, equalizer bars on the playing TrackRow,
the progress-scrub flourish, sidebar slide-in, screen crossfades — must
respect Reduce Motion.

```swift
@Environment(\.accessibilityReduceMotion) private var reduceMotion

.animation(reduceMotion ? nil : .spring(response: 0.35), value: screen)
```

For purely ambient / decorative loops (vinyl spin, eq bars), swap to a
static frame when `reduceMotion == true`.

**Acceptance criteria**
- System Settings > Accessibility > Display > Reduce Motion → on.
- Relaunch Jellify. No looping animation plays, no spring transitions,
  screen swaps are instant crossfades (or fade-free).
- Progress bar still updates (it's not an animation, it's state).

### Issue 11: Increase Contrast variants of the theme
**Labels:** `area:macos`, `area:a11y`, `kind:feat`, `priority:p1`
**Effort:** M

WCAG 2.2 AA requires 4.5:1 for ≤18 pt text and 3:1 for ≥18 pt / bold. Quick
check of current tokens:

| Token | On bg #0C0622 | On bgAlt #140B30 | Pass? |
| --- | --- | --- | --- |
| ink #FFFFFF | 19.2:1 | 18.1:1 | yes |
| ink2 #7E72AF | 4.9:1 | 4.6:1 | marginal |
| ink3 alpha .65 | ≈3.1:1 | ≈2.9:1 | **fails at small text** |
| accent #CC2F71 | 3.4:1 | 3.2:1 | fails for body copy |

Actions:
1. Add `Theme.highContrast` tokens. When
   `@Environment(\.colorSchemeContrast) == .increased`, swap `ink3` to
   at least `rgba(198, 190, 220, 1)` (7+:1) and accent to `accentHot`
   (#FF066F, 6.7:1 on bg).
2. Provide an `AccessibleTheme` wrapper:
   ```swift
   @Environment(\.colorSchemeContrast) var contrast
   var ink3: Color { contrast == .increased ? Theme.ink3HC : Theme.ink3 }
   ```
3. Replace `Theme.border` (alpha .18 → ≈1.4:1, invisible with contrast
   on) with a non-alpha HC variant.

**Acceptance criteria**
- System Settings > Accessibility > Display > Increase Contrast → on.
- Every body-weight `Text` on every surface passes 4.5:1 when measured
  with Colour Contrast Analyser or Xcode Accessibility Inspector's
  Contrast audit.
- Borders and dividers become visibly solid, no longer alpha-blended.

### Issue 13: Accessibility announcements for player lifecycle events
**Labels:** `area:macos`, `area:a11y`, `kind:feat`, `priority:p1`
**Effort:** S

When a new track starts (either via user action or autoplay), post an
`AccessibilityNotification.Announcement` so VO users hear it without
needing to navigate back to the player bar.

```swift
import AppKit

func announceTrackChange(to track: Track) {
    let msg = "Now playing \(track.name) by \(track.artistName)"
    NSAccessibility.post(
        element: NSApp.mainWindow as Any,
        notification: .announcementRequested,
        userInfo: [
            .announcement: msg,
            .priority: NSAccessibilityPriorityLevel.high.rawValue,
        ]
    )
}
```

Hook it from `AppModel` when `status.currentTrack` changes (use a
`didSet` or `onChange(of:)` observer at the top of `MainShell.swift`).

**Acceptance criteria**
- VO on, skip to next track with ⌘→ (or the forward button). VO speaks
  "Now playing Clair de Lune by Claude Debussy" without interrupting the
  current VO focus context.
- Announcements throttle: rapid next/next/next emits only the final one
  (debounce by 300 ms).

### Issue 18: Figtree glyph coverage fallback chain
**Labels:** `area:macos`, `area:i18n`, `kind:feat`, `priority:p0`
**Effort:** M

Figtree covers Latin Extended (enough for en/es/fr/de/it/pt-BR) but has no
Cyrillic, Greek, Hebrew, Arabic, Thai, CJK. When a Russian or Japanese
string is rendered with `Font.custom("Figtree-Regular", …)`, macOS will
fall back — usually to .AppleSystemUIFont — but metrics and weight won't
match, giving a broken ransom-note look. Use a composite font with CoreText:

```swift
private static func fallbackFont(baseName: String,
                                 size: CGFloat,
                                 weight: Font.Weight) -> Font {
    let baseDesc = CTFontDescriptorCreateWithNameAndSize(baseName as CFString, size)
    let fallback = CTFontDescriptorCreateCopyWithAttributes(
        baseDesc,
        [kCTFontCascadeListAttribute: [
            // Apple's system fonts with the script coverage we need
            CTFontDescriptorCreateWithNameAndSize("SFProText-Regular" as CFString, size),
            CTFontDescriptorCreateWithNameAndSize("PingFangSC-Regular" as CFString, size),
            CTFontDescriptorCreateWithNameAndSize("HiraginoSans-W3" as CFString, size),
        ]] as CFDictionary
    )
    // Build Font from CTFont
}
```

Alternatively simpler: conditionally pick system font based on locale:

```swift
private static func preferredFontName(for weight: Font.Weight) -> String {
    let locale = Locale.current.language.languageCode?.identifier ?? "en"
    if ["ru", "uk", "bg", "el", "ja", "zh", "ko", "ar", "he", "th"].contains(locale) {
        return systemFontName(for: weight)  // fall back entirely to system
    }
    return figtreeFontName(for: weight)
}
```

**Acceptance criteria**
- Switch system language to Japanese. The UI renders in Hiragino Sans /
  SF Pro, no tofu boxes.
- Switch to Russian. Cyrillic characters render correctly at all weights.
- Mixed English+Japanese string ("Now playing 君の知らない物語") renders both
  scripts readably.

### Issue 20: Locale-correct dates, numbers, durations
**Labels:** `area:macos`, `area:i18n`, `kind:polish`, `priority:p1`
**Effort:** M

Replace every hand-built format string with `FormatStyle`:

- Dates: `Date.FormatStyle.relative(presentation: .named)` for "2 hours ago"
  — Apple translates into every locale.
- Numbers: `Int.formatted()` — auto thousands separator (1,234,567 / en;
  1.234.567 / de; 1 234 567 / fr; 1,234,567 / ja).
- Durations: `Duration.TimeFormatStyle(pattern: .minuteSecond)` for the
  classic `3:42`. For a11y announcements prefer the spelled-out form via
  `DateComponentsFormatter.unitsStyle = .spellOut`. Keep the visible
  "3:42" but the accessibilityValue uses the spoken form.
- Time zones: server timestamps arrive in UTC from Jellyfin. Always convert
  with `TimeZone.current` for display. Do not hand-parse.

Replacements hotlist:
- `PlayerBar.swift:73` and `:91` — `format(positionSeconds)` and
  `format(durationSeconds)` — audit the current `format` helper, likely a
  custom "%d:%02d" that will be wrong in RTL and miss pluralization.
- `TrackRow.swift` duration column
- `AlbumDetailView.swift` album year / total runtime
- `Sidebar.swift:58` — "Connected · N albums" — see Issue 21.

**Acceptance criteria**

### Issue 21: Pluralization via `.stringsdict` / xcstrings variations
**Labels:** `area:macos`, `area:i18n`, `kind:feat`, `priority:p1`
**Effort:** M

"N albums" must pluralize correctly for every locale. English has 2 forms,
Russian has 4 (one/few/many/other), Arabic has 6, Polish has 3.
`Localizable.xcstrings` supports this natively — each key can have a
"Plural" variation with cases `zero`, `one`, `two`, `few`, `many`, `other`.

```swift
Text("^[\(model.albums.count) album](inflect: true)")
```

Or explicitly with `String.LocalizationValue`:

```swift
Text("\(model.albums.count) albums", comment: "Sidebar album count")
// in xcstrings, set this key to Plural:
// one → "%lld album", other → "%lld albums"
```

Hotlist for pluralization:
- Album count, artist count, track count, played count
- Empty states ("No results" / "1 result" / "N results")
- Queue length ("Queue: 1 track" / "Queue: N tracks")
- Duration when spelled out ("1 second" / "N seconds")

**Acceptance criteria**
- Switch to Russian. Sidebar reads "1 альбом", "2 альбома", "5 альбомов",
  "21 альбом". Our pluralization catalog handles every case.
- Unit tests in `macos/Tests/LocalizationTests/PluralsTests.swift` assert
  each locale's output for counts 0, 1, 2, 5, 11, 21, 101.

### Issue 22: Localize error messages (not from Rust core)
**Labels:** `area:macos`, `area:i18n`, `kind:feat`, `priority:p0`
**Effort:** M

## Shipping checklist

Before a release can be marketed as "accessible + localized":

1. **Xcode Accessibility Inspector Audit — 0 errors, 0 warnings** on every
   screen (Login, Library, Search, Album Detail, Player idle, Player
   playing, Sidebar at all Dynamic Type sizes). Evidence archived to
   `macos/docs/a11y/audits/<release-tag>/`.
2. **VoiceOver end-to-end walkthrough** completed by one sighted developer
   plus one blind/low-vision tester. Recording saved.
3. **Voice Control and Switch Control** tested against transport and
   navigation flows.
4. **Dynamic Type at +5** — layouts don't clip; screenshots in repo.
5. **Reduce Motion, Reduce Transparency, Increase Contrast, Differentiate
   Without Color, Invert Colors** all toggled and the app stays functional.
6. **WCAG 2.2 AA contrast** verified on every text color / surface pair with
   Colour Contrast Analyser. Report pasted into the release PR.
7. **Color-blindness sim screenshots** for all three themes across the
   three major vision types.
8. **All 9 locales (en + 8 first-wave)** render without missing keys; CI's
   `lint:i18n` job is green.
9. **Pluralization unit tests** pass for counts 0, 1, 2, 5, 11, 21, 101 in
   every wired locale.
10. **RTL preview** — Arabic environment renders mirrored without clipping;
    screenshots in repo.
11. **Font fallback** — Japanese and Russian UI strings render in a system
    font chain, not as tofu or fallback-mismatched weights.
12. **Error messages** from the Rust core are all Swift-mapped; no English
    strings cross the FFI.
13. **Menu bar** — every command item is localized, labeled for VO, has a
    keyboard shortcut where appropriate, and no shortcut conflicts with a
    system-reserved combo in any supported locale.
14. **AccessibilityNutritionLabel.md** filled in with evidence PRs for every
    claimed badge.
15. **Regression tests** for accessibility labels pass in CI.
